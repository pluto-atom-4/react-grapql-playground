WHY THIS SEQUENCING
================================================================================

Core Principle: Backend Contract → Frontend Implementation → Integration

Phase 1 (Backend) defines:
  ✓ JWT format and validation logic
  ✓ context.user shape and availability
  ✓ Required Authorization header format
  ✓ Error codes for unauthenticated requests

Phase 2-3 (Frontend) build against this contract:
  ✓ Form injects token in correct format
  ✓ Apollo link injects Authorization header
  ✓ Login/logout flow uses defined context shape

Phase 4 (Integration) validates everything works together:
  ✓ Token format is correct
  ✓ Header injection works
  ✓ Resolvers accept and process token
  ✓ All layers coordinate without surprises

Without this sequencing:
  ✗ Frontend guesses JWT format
  ✗ Backend implements different format
  ✗ Integration fails
  ✗ Rework both layers = 2-3 days added

With this orchestration:
  ✅ 45-min wait for contract is cheaper than 2-3 days of rework

3 CRITICAL PATTERNS (Anti-Patterns Prevented)
================================================================================

Pattern 1: Fresh-Per-Request Tokens
─────────────────────────────────────
✓ const { token } = useAuth()  // Always current
✗ let cachedToken = null        // Stale!

Why: If token expires/refreshes, closures have stale values.

Pattern 2: Protected Resolver Guards
─────────────────────────────────────
✓ Query.builds = (_, __, { user }) => {
    if (!user) throw AuthenticationError(...)
    return prisma.build.findMany(...)
  }

✗ Query.builds = (_, __, { user }) => {
    return prisma.build.findMany()  // Assumes middleware did it all
  }

Why: Defensive. Middleware might be bypassed in tests/debugging.

Pattern 3: Cache Reset on Logout
──────────────────────────────────
✓ client.cache.reset()  // Clear cached user data
  logout()              // Clear token
  router.push('/login') // Redirect

✗ logout()
  router.push('/login')  // Cache still has old user data!

Why: Apollo cache survives token change. Security leak if not reset!

================================================================================
INTERVIEW TALKING POINTS (5 Key Messages)
================================================================================

1. "Why This Sequence?"
   Backend is the contract. Phase 1 defines what the frontend must do.
   If we parallelize without the contract, frontend makes assumptions
   Phase 1 changes. Integration fails hard. Better to wait 45 min than
   rework 3 days.

2. "How Prevent Common Auth Bugs?"
   Three patterns: fresh-per-request tokens (not stale closures),
   protected resolver guards (defensive validation), cache reset on
   logout (prevent data leakage). These prevent 80% of auth bugs.

3. "Why localStorage instead of cookies?"
   MVP: simplicity. Production: httpOnly cookies (immune to XSS).
   Architecture doesn't change—AuthContext still provides token,
   just fetched differently.

4. "What if developer disagrees?"
   I'd ask which phase they'd start with. If they want to parallelize,
   I'd say Phase 2 is hard-blocked by Phase 1 (needs JWT format known).
   Getting agreement on hard blockers surfaces miscommunication early.

5. "How handle distributed team?"
   Phase 1 becomes async. Backend publishes spec (JWT logic, context
   shape, headers). Frontend reviews, asks questions. Once locked,
   frontend implements independently. Only integrate synchronously.
