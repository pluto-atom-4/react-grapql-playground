                        ARCHITECTURE PATTERN
================================================================================

The project uses a DUAL-STRATEGY APPROACH:

┌─────────────────────────────────────────────────────────────────────────┐
│ SERVER LAYER (Per-Request)                                             │
│ ├─ Pattern: registerApolloClient (Apollo integration)                  │
│ ├─ Cache: Fresh InMemoryCache per HTTP request                         │
│ ├─ Security: Prevents cache pollution (Issue #107 FIXED)              │
│ └─ Use Case: Server Components, initial data fetch                     │
└─────────────────────────────────────────────────────────────────────────┘
                                  ↓
┌─────────────────────────────────────────────────────────────────────────┐
│ CLIENT LAYER (Singleton)                                               │
│ ├─ Pattern: makeClient() wrapped in useMemo                            │
│ ├─ Cache: Persistent across renders                                    │
│ ├─ Performance: Optimistic updates, real-time SSE (Issue #23 FIXED)   │
│ └─ Use Case: Client Components, mutations, interactivity              │
└─────────────────────────────────────────────────────────────────────────┘

Why Two Configurations?
- Server: Each request must have isolated cache (multi-user, security)
- Client: Single user needs persistent cache (state, performance)

================================================================================
                         KEY FINDINGS
================================================================================

DUAL-STRATEGY PATTERN:

Server Components (registerApolloClient):
✅ Use registerApolloClient pattern
✅ Fresh cache per request (automatic)
✅ Pass data as props to Client Components
✅ Fetch all data before rendering
❌ Don't use hooks
❌ Don't create multiple clients per request

Client Components (makeClient with useMemo):
✅ Use useMemo wrapper for singleton
✅ Persistent cache across renders
✅ Mutations with optimistic updates
✅ Real-time subscriptions via SSE
❌ Don't call makeClient() without useMemo
❌ Don't recreate client on each render

PERFORMANCE STRATEGY:

Data Hydration (Critical):
- Server fetches initial data
- Passes as props to Client Component
- Eliminates double-fetch on mount
- Faster perceived performance

Why Not Client-Side Only?
❌ Double-fetch (server renders, client fetches)
❌ Loading spinner after SSR (worse UX)
❌ Wastes bandwidth
❌ Slower perceived performance

CACHE MANAGEMENT:

Server Cache:
├─ Fresh per request
├─ Automatically cleaned up
├─ No manual lifecycle
└─ No cross-request contamination

Client Cache:
├─ Singleton (persists session)
├─ Updated by mutations immediately
├─ Real-time events invalidate/update
└─ Lives until page reload

                       BEST PRACTICES IDENTIFIED
================================================================================

SERVER COMPONENTS (5 DO's, 2 DON'Ts):

✅ DO: Use registerApolloClient
   const client = getClient()

✅ DO: Fresh cache per request  
   (automatic with registerApolloClient)

✅ DO: Pass data as props
   <BuildsList initialBuilds={data.builds} />

✅ DO: Fetch all data before rendering
   const { data } = await client.query(...)

❌ DON'T: Use hooks in Server Components
   useQuery() won't work

❌ DON'T: Create multiple clients per request
   Reuse single client instance


CLIENT COMPONENTS (5 DO's, 2 DON'Ts):

✅ DO: Use useMemo for singleton
   const client = useMemo(() => makeClient(), [])

✅ DO: Persistent cache across renders
   Cache stays alive during component lifecycle

✅ DO: Mutations with optimistic updates
   useMutation(..., { optimisticResponse: {...} })

✅ DO: Real-time subscriptions
   useEffect(() => { eventSource.addEventListener(...) })

❌ DON'T: Call makeClient() without useMemo
   Creates new instance on every render

❌ DON'T: Recreate clients
   Cache lost on re-renders


COMMON PITFALLS (4 documented):

1. Using hooks in Server Components
   Problem: Hooks only work in Client Components
   Solution: Use async/await with getClient()

2. makeClient() without useMemo
   Problem: New client on each render = lost cache
   Solution: Wrap in useMemo with empty dependency array

3. Double-fetching data
   Problem: Server fetches, then Client fetches again
   Solution: Pass initial data as props

4. Forgetting hydration
   Problem: Client cache starts empty
   Solution: Initialize Apollo cache with server data

ISSUE #23 - Apollo Client Singleton
Status: ✅ FIXED
Problem: makeClient() called on every render
Solution: useMemo wrapper in apollo-wrapper.tsx

ISSUE #107 - Cache Pollution
Status: ✅ PREVENTED

Solution: registerApolloClient per-request isolation
Result: Each request gets fresh cache (secure)

                    INTERVIEW PREPARATION VALUE
================================================================================

Talking Points Ready:

1. "Why two Apollo configurations?"
   Answer: Server-side per-request isolation prevents cache pollution (Issue 
   #107). Client-side singleton maintains state across renders (Issue #23).

2. "How does data flow from Server to Client?"
   Answer: Server Component fetches initial data with getClient(), passes as 
   props to Client Component. Eliminates double-fetch on hydration.

3. "How do optimistic updates work?"
   Answer: Mutation shows instant feedback before server confirms. Apollo 
   cache updates immediately, then server response confirms or rolls back.

4. "Why is registerApolloClient important?"
   Answer: Prevents cache pollution where User A's data could leak to User B. 
   Official Apollo pattern for Next.js App Router.

5. "How does real-time work with Apollo?"
   Answer: Server emits SSE events, client listener invalidates or updates 
   Apollo cache. Cache changes trigger component re-renders automatically.
