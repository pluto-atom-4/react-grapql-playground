PREVENTION STRATEGIES FOR PHASE 2-4
═══════════════════════════════════════════════════════════════════════════

STRATEGY 1: Component Registry (RECOMMENDED)
─────────────────────────────────────────────
Create docs/COMPONENT-REGISTRY.md to track:
  • Which components are "reserved" by which issues
  • Expected API contracts for new components
  • File ownership and dependencies

Benefits:
  ✓ Clear ownership of components
  ✓ Prevents duplicate implementations
  ✓ Enables early coordination
  ✓ Single source of truth for file reservations

Implementation:
  1. Before starting work: check COMPONENT-REGISTRY.md
  2. Add entry: "Issue #XXX: Creating [components]"
  3. Post GitHub comment: "Starting work on [files]"
  4. Wait for team acknowledgment
  5. Begin development only after confirmation

STRATEGY 2: Branch Synchronization Workflow
────────────────────────────────────────────
Add to every developer's workflow:

Daily/Every 2-3 days:
  git fetch origin
  git rebase origin/main

Before requesting review:
  git merge --no-commit --no-ff origin/main  # Test for conflicts
  git reset --hard HEAD                       # Undo test
  git push --force-with-lease origin [branch]

Benefits:
  ✓ Early conflict detection
  ✓ Fixes conflicts incrementally (not all at once)
  ✓ Reduces rebase pain
  ✓ Keeps PR small and focused

STRATEGY 3: Dependency Tracking
───────────────────────────────
Add to every Phase 2-4 issue description:

## Dependencies
This issue depends on:
- Issue #ABC (must complete first)
- Issue #XYZ (needed for this feature)

This issue blocks:
- Issue #LMN (waiting for this feature)

## File Reservations
Files I will create/modify:
- frontend/components/NewFeature.tsx (new)

Files I will NOT touch:
- FormComponents/* (reserved for #257)
- StatusBadge.tsx (reserved for #255)

Benefits:
  ✓ Clear dependency chain
  ✓ Coordination awareness
  ✓ Prevents accidental file conflicts
  ✓ Enables proper PR ordering

STRATEGY 4: Pre-Merge Conflict Verification
────────────────────────────────────────────
Add to PR template:

## Conflict Prevention Checklist
- [ ] I have rebased my branch on latest main
- [ ] I have tested for conflicts: `git merge --no-commit --no-ff main`
- [ ] No conflicts found OR conflicts documented
- [ ] If conflicts exist, I have coordinated with teams

## Component Reservations
Affected components:
- [ ] FormComponents (reserved by #257)
- [ ] StatusBadge (reserved by #255)
- [ ] Other component: [specify]

Benefits:
  ✓ Early detection before merge
  ✓ Forces conflict testing
  ✓ Documents conflict coordination
  ✓ Accountability for developers

STRATEGY 5: Weekly Team Sync Meeting
─────────────────────────────────────
Every Monday during Phase 2-4:

Agenda (30 minutes):
  1. What's currently in-flight? (5 min)
  2. Are any two issues touching same files? (5 min)
  3. Do developers know about each other's work? (5 min)
  4. Any blocked issues? (5 min)
  5. Update COMPONENT-REGISTRY.md (5 min)

Output:
  ✓ Identified conflicts before merge
  ✓ Rebalanced work if needed
  ✓ Updated component registry
  ✓ Improved team coordination

Benefits:
  ✓ Early detection of conflicts
  ✓ Enables proactive coordination
  ✓ Improves team awareness
  ✓ Reduces context switching
